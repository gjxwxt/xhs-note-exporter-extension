# XHS Note Capture

## 背景

这个扩展是为小红书网页版的帖子采集场景做的。

目标不是全自动滚动和全自动点开账号下所有帖子，而是：

- 用户自己正常浏览和打开帖子详情页
- 扩展稳定采集帖子核心信息
- 把多条记录汇总到同一个任务里
- 最终导出为适合后续分析的 Excel

最初思路尝试过“自动滚动列表 + 自动点开帖子 + 自动抓取”，但在小红书页面上容易遇到：

- DOM 结构频繁变化
- 弹层状态不稳定
- 安全验证 / 风控
- 自动化链路过长后，单点失败会让整轮任务卡死

因此当前版本的核心策略是：

- 不自动滚动账号列表
- 不自动代替用户点开帖子
- 以“用户打开帖子详情页”为稳定输入
- 在这个前提上做自动采集、去重、导出

## 用户需求摘要

用户要做的是：

- 针对某个小红书账号的帖子做内容盘点
- 采集封面、标题、正文、点赞、评论、发布时间等字段
- 汇总到 Excel
- 后续再和销量等业务数据做关联分析

用户非常在意的点：

- 尽量少做重复点击
- 采集过程能感知进度或结果
- Excel 里封面图可用
- 失败时不能静默
- 数据准确性优先于“全自动”

## 当前产品形态

当前扩展支持两种采集方式：

1. 手动采集
2. 自动采集

### 手动采集

- 用户打开帖子详情页
- 点击 popup 中的“采集当前帖子”
- 当前帖子会写入已选中的采集任务

### 自动采集

- 用户先在 popup 中选择采集任务，并开启“自动采集”
- 后续只要用户打开新的帖子详情页
- content script 会监听路由变化并自动尝试采集
- 如果是重复帖子，会自动跳过并弹 toast

## 关键架构

### 1. `background.js`

职责：

- 维护采集任务状态
- 接收 popup / content script 的消息
- 去重、删除、清空、删除任务
- 生成 Excel
- 下载封面图
- 汇总最近异常

关键状态存储在 `chrome.storage.local`：

- `xhsManualCaptureTasks`
- `xhsManualActiveTaskId`
- `xhsAutoCaptureEnabled`
- `xhsRecentErrors`

### 2. `content.js`

职责：

- 在小红书页面内读取帖子详情
- 监听路由变化
- 自动采集时弹页面 toast
- 处理详情页 DOM 和 `__INITIAL_STATE__`
- 做页面侧异常提示和错误上报

### 3. `popup.js`

职责：

- 任务选择 / 创建
- 开关自动采集
- 手动触发采集
- 清空任务 / 删除任务 / 删除单条
- 高清导出 / 快速导出
- 展示最近采集记录和最近异常

### 4. `popup.html` / `popup.css`

职责：

- popup 界面结构和样式
- 目前强调信息密度和操作效率，不追求复杂流程

## 采集字段

Excel 明细当前字段为：

- `封面图`
- `序号`
- `账号/昵称`
- `发布时间`
- `标题`
- `正文`
- `点赞数`
- `评论数`
- `笔记类型`
- `笔记链接`
- `抓取状态`
- `备注`

说明：

- 过去这里叫 `小红书号`
- 但很多页面拿不到稳定的 `redId`
- 所以当前导出字段已经调整为 `账号/昵称`
- 优先用 `redId`
- 拿不到时回退到作者昵称

## 数据读取策略

这是本项目最重要的历史决策之一。

### 时间 / note payload

优先来源：

- `window.__INITIAL_STATE__.note.noteDetailMap[*].note`

原因：

- 页面上经常只展示相对时间，例如“4小时前”“编辑于 03-16”
- 内嵌状态里通常有更精确的时间戳，如 `time`、`lastUpdateTime`

当前实现会：

- 优先读原始时间戳
- 再回退到 DOM 文本
- 对“昨天 / 前天 / 几小时前 / 03-16”这类文本做规范化

### 点赞 / 评论

这里做过一次关键修正。

经验结论：

- `__INITIAL_STATE__` 里的互动数据通常可用
- 但某些页面用户更信任页面上真实展示的数字
- 评论 DOM 不是固定 `comment-wrapper`，有时实际是 `chat-wrapper`

所以当前优先级是：

1. 详情页底部互动栏 DOM
2. `interactInfo`
3. 其它深层 fallback

其中底部互动栏会定点查找：

- `.like-wrapper .count`
- `.chat-wrapper .count`
- `.comment-wrapper .count`

并且限制在详情页的互动区根节点里查找，避免误抓到列表页的其它 `.count`

### 封面图

优先来源：

1. `note.imageList`
2. 详情页媒体区里索引为 `0` 的首图 DOM
3. 深层图 URL fallback

当前已做过一次导出质量调整：

- 早期版本把图缩得太小，Excel 里显得糊
- 现在提高了图片边长上限
- 并且如果原图已经是合适的 `jpeg/png`，会直接复用，减少重编码

当前还额外约束了两点：

- 多图帖如果内嵌状态可用，强制以 `imageList[0]` 作为封面
- 视频帖当前默认不采集封面，避免误抓播放器 poster 或评论区图片

## 导出策略

当前支持两种导出：

### 高清导出

- 带封面图
- 更适合最终留档或少量精细分析
- 耗时主要在图片下载和嵌入

### 快速导出

- 不带封面图
- 适合大批量记录，例如 100 到 200 条
- 明显更快，也更稳

## 去重策略

当前去重规则：

1. 优先按 `noteUrl`
2. 如果 `noteUrl` 缺失，则退化为：
   `账号/昵称 + 发布时间 + 标题`

自动采集遇到重复：

- 不新增
- 不覆盖
- 直接跳过
- 页面会弹 toast 提示

手动采集遇到重复：

- 会更新原记录

## 当前 UI 行为约定

### popup 列表

- “已采集标题”区域最多显示约三条高度
- 内部滚动
- 每条支持删除

### destructive 操作

这里有一个历史坑：

- 之前 popup 里用过 `window.confirm`
- 在扩展 popup 里这个交互不稳定，用户会感觉“按钮没反应”

所以现在：

- 删除单条：直接执行
- 清空任务：直接执行
- 删除当前任务：直接执行

如需二次确认，后续建议做成 popup 内自己的确认面板，而不是再用浏览器原生 confirm

## 异常处理

当前已经有“最近异常”链路：

- background 维护最近异常列表
- popup 显示最近一条异常摘要
- content script 发生异常时会弹页面 toast，并上报到 background
- background 本身也监听全局错误和未处理 Promise 拒绝

注意：

- 这里的目标是“不静默失败”
- 不是做完整监控平台

## 已知限制

### 1. 不负责自动点开帖子

这是故意保留的约束。

理由：

- 用户手动点开帖子最稳定
- 自动点开 + 自动关闭容易撞风控
- 弹层和路由状态比单页采集更脆弱

### 2. DOM 依赖仍然存在

尽管已经尽量优先读内嵌状态，但：

- 点赞 / 评论 / 作者展示名
- 某些边缘字段

仍然依赖 DOM fallback。小红书页面结构一旦大改，需要重新适配。

### 3. 大批量高清导出仍然慢

哪怕做了快速导出和原图复用：

- 100+ 条带封面图导出
- 仍然会受网络、图片大小、浏览器内存影响

若未来还要继续优化，优先级建议是：

1. 分批导出
2. 后台并发限速下载图片
3. 可选图片压缩档位

## 给下一个接手者的建议

如果要继续做下一步优化，优先顺序建议是：

1. 真机验证删除单条 / 清空任务 / 删除任务 / 快速导出
2. 用不同类型帖子验证作者昵称和互动数
3. 评估是否需要“分批导出”
4. 只有在用户强需求下，再考虑自动打开帖子链路

不要轻易把当前方案重新推回“全自动滚动 + 全自动点帖”的老路线。

那条路线在这个站点上不稳，历史上已经被证明维护成本很高。
